Snackable Games — Game Content Creation Guidelines

For game content creators building H5 / Three.js / Phaser games for the Snackable Games platform.

1. Overview

Snackable Games are short, web-based games (HTML5 / JS / CSS) that play inside a sandboxed WebView on Meta products (e.g., Horizon mobile). Your game ships as a single self-contained HTML file served from a sandboxed domain (embed.fbsbx.com) with a strict Content Security Policy (CSP). All assets must be packaged with the game at build time — there is no network at runtime.

1.1 Incompatible Formats — Do NOT Submit

IMPORTANT: The following formats cannot be hosted on the Snackable Games platform. If your game was built using any of these technologies, it must be rebuilt as a web-compatible HTML/JavaScript game before submission.

Format / Technology

Why It Doesn't Work

Horizon Worlds projects (meta/worlds SDK)

Proprietary runtime with no browser support. The meta/worlds ECS, scene format, and APIs cannot execute in a WebView.

.hstf files (Horizon scene/template)

Proprietary binary format. No browser parser exists.

.material files (Horizon materials)

Reference assets by internal platform IDs — not self-contained.

.xaml files (Noesis XAML UI)

Requires the Noesis UI renderer. Use HTML/CSS for game UI instead.

.pkfx / .pkma files (PopcornFX)

Proprietary particle effects middleware with no browser runtime.

.cso files (compiled DirectX shaders)

DirectX HLSL bytecode — cannot run in WebGL. Use GLSL shaders instead.

.dds files (DirectX textures)

DirectX Block Compression format. Convert to PNG or WebP.

.surface files (Horizon custom shaders)

Proprietary shader format. Use Three.js ShaderMaterial with standard GLSL.

If you have a game built in Horizon Worlds / Meta Horizon Studio: The game logic must be rewritten using standard web technologies (Three.js, Phaser, or vanilla Canvas/WebGL). Web-compatible assets (PNG, MP3, GLB) can be reused directly, but all meta/worlds SDK code, XAML UI, and proprietary scene files must be replaced.

3. File Structure

Please place your code under fbsource repo

Each game lives in its own directory at nest/apps/snackables/public/games/<your_game_name>/:

File

Description

index.html

Required. The game entry point.

index_autostart.html

Optional. Autostart variant that skips "tap to start" (see §8).

assets/, *.js, *.css

Your game code, styles, images, audio, models, fonts. Filenames are arbitrary.

Rules:

  • The directory must contain index.html.
  • To ship an autostart variant (skips "tap to start"), add an index_autostart.html file in the same directory.
  • All assets the game needs must live inside this directory (use relative paths).
  • The game must be expressible as a single self-contained HTML file — there is no file system at runtime, so everything gets inlined at build time.

Size limits

  • 2 MB target per file. Bigger sizes come with latency tradeoffs — keep assets small.
  • 5 MB hard cap on the final inlined HTML. Ideally, keep games simple and small — under 2 MB is the target. The pipeline will reject games exceeding 5 MB.
  • Well-optimized games stay under 2 MB. If yours is larger, optimize textures, compress audio to low-bitrate MP3, and reduce model polygon counts.

GLB textures (automatic optimization)

GLB textures are auto-recompressed: textures are downscaled to 512 px max and converted to WebP (PNG alpha preserved). Typical reduction: 65–84% smaller files. Author your textures generously — the pipeline will optimize them. Pre-optimize if your raw GLBs combined exceed 2 MB. Target under 500 KB per model after compression.

Google Fonts

Reference Google Fonts via <link rel="stylesheet"> — the CSS and woff2 files are downloaded and embedded for you.

5. Supported Browser APIs

Games run in a sandboxed iframe on embed.fbsbx.com with strict CSP. These APIs work:

  • DOM — full read/write
  • Canvas 2D, WebGL, WebGL2
  • Web Audio, <audio>, new Audio(...)
  • <video> / <source>
  • requestAnimationFrame, setInterval, setTimeout
  • Touch / pointer / keyboard events
  • localStorage / sessionStorage (per-session only — see Restrictions)

6. Restrictions (Blocked by CSP)

Your game cannot:

  • Make any network calls — fetch(), XMLHttpRequest, WebSocket are all blocked (connect-src: 'none'). Everything must be inlined at build time.
  • Load <script src="..."> from arbitrary CDNs — only the supported library CDNs (jsdelivr, unpkg, cdnjs, esm.sh) are rewritten; everything else is blocked.
  • Use <iframe> inside the game.
  • Use inline event handlers like onclick="...", onload="..." in HTML — use addEventListener instead.
  • Call requestFullscreen() — fullscreen is intentionally disabled.
  • Use DRACOLoader — its remote WASM decoder is blocked. If your model uses Draco compression, decompress it ahead of time.
  • Rely on persistent storage between plays — treat each play as a fresh session.
  • Depend on createImageBitmap for Three.js GLB textures — the pipeline routes through TextureLoader instead.

7. Dos and Don'ts

DO

  • Use one of the supported library versions in §2.
  • Reference local assets with relative paths — they'll be auto-inlined.
  • Use Three.js TextureLoader for GLB textures.
  • Use standard patterns: <img src=...>, new Audio(...), <video><source src=...>, Phaser this.load.*().
  • Use Google Fonts via <link rel="stylesheet">.
  • Use addEventListener for events.
  • Respect device safe areas — read the bottomInset URL query parameter (a numeric pixel value; add as bottom padding to your UI container) to avoid colliding with the iPhone home indicator.
  • Test locally before shipping (see §8).

DON'T

  • Don't call fetch, XMLHttpRequest, or open WebSocket connections.
  • Don't load scripts from random CDNs.
  • Don't reference DRACOLoader.
  • Don't use inline onclick=, onload=, etc.
  • Don't embed <iframe> inside your game.
  • Don't call requestFullscreen().
  • Don't ship raw asset files larger than 5 MB.
  • Don't rely on persistent storage.
  • Don't render "tap to start" menus on <canvas> if you want the autostart variant to work — the autostart shim only handles DOM overlays.
  • Don't leave Android's soft keyboard up after hiding an input — call .blur() explicitly.

8. Testing Your Game

Local preview

phps SnackableGamesPreprocessorScript \

  nest/apps/snackables/public/games/<your_game> --local

cd ~/fbsource/nest/apps/snackables && yarn dev

# open the URL the dev server prints

Useful flags

About the autostart variant: If your game has a "tap to start" screen, you can ship an autostart variant that skips directly into gameplay. To do this, create a file named index_autostart.html alongside your index.html. The build pipeline adds a shim that auto-dismisses DOM-based start overlays. Note: if your start screen is rendered on <canvas> rather than as a DOM overlay, the autostart shim won't work — you'll need to bypass the menu state in your game code.

  • --autostart — preview the autostart (skip "tap to start") variant.
  • --dry-run — classify and validate only; no transformation.
  • --skip-fonts — skip Google Fonts inlining for faster iteration.
  • --batch — preview every game in the directory.

End-to-end production validation

Use Bloks Playground → Horizon Playable Video → Snackable Games List. It lists every shipped game with PLAYABLE / BROKEN status, resolves a fresh signed URL on demand, and launches the game in a real WebView. Use this to confirm your game works end-to-end after upload.

What the preprocessor checks for you

  • CSP compliance — flags inline event handlers, iframes, and fetch/XHR/WebSocket usage.
  • External library validation — verifies external scripts are on the supported CDN allowlist.
  • Asset audit — diagnostic scan for un-inlined references.
